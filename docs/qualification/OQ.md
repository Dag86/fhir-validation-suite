## 1. Purpose

This Operational Qualification document verifies that each tool in the FHIR R4 API Validation Suite toolchain performs as intended against known inputs and expected outputs. OQ tests are designed so the correct result is known in advance — the tool itself is being tested, not the FHIR server.

OQ is the second phase of tool qualification. IQ must be completed and passed before OQ begins. OQ must be completed and passed before Performance Qualification (PQ) begins.

**The critical principle of OQ:** If a tool produces a passing result when the correct result is a failure, all validation evidence generated by that tool is invalid. Every OQ test that checks failure detection is as important as every OQ test that checks pass detection.

---

## 2. OQ Qualification Risk Assessment

| Tool | Highest Risk OQ Failure Mode | Consequence |
|---|---|---|
| Git | Commit does not create a retrievable, attributable record — or commit SHA is not linked to pipeline evidence | Validation evidence cannot be traced to a specific, known source state — evidence chain is broken |
| Karate DSL | Failing assertion reported as passing | False compliance evidence — invalid validation package |
| HL7 FHIR Validator | Invalid resource reported as valid | False conformance evidence — invalid validation package |
| GitHub Actions | Reports not archived or timestamps incorrect | Validation evidence not retrievable for audit |

## 3. OQ — Git

**Objective:** Confirm Git correctly creates attributable, retrievable commit records that form an auditable evidence chain, and that the pull request workflow enforces review before changes reach `main`.

**Precondition:** IQ-GIT all steps passed.

**Rationale:** The regulatory value of Git as a document control system depends entirely on its audit trail functioning correctly. If commits are not correctly attributed, timestamped, and linked to CI pipeline runs, the entire evidence chain is undermined. These OQ steps verify Git's core audit trail functions against known inputs.

## 7. Negative Control Interpretation

### Purpose

Per GAMP 5 §10.3 and FDA General Principles of Software Validation
§4.5, a validation suite must demonstrate failure detection capability.
Negative controls verify that the test framework correctly identifies
non-conforming inputs and does not produce false-negative results.
A suite that only produces green results is not a validated detection
system — it may be passing because it cannot detect failures.

### Negative Control Results

| Step ID | Control Type | Expected Framework Response | Actual Response | OQ Step Result |
|---|---|---|---|---|
| OQ-KAR-002 | Known false assertion (1 == 2) | Karate reports FAIL | FAIL — match failed: EQUALS | PASS |
| OQ-KAR-004 | Schema match with missing required field | Karate reports FAIL | FAIL — match failed: EQUALS | PASS |
| OQ-VAL-002 | Non-conformant FHIR resource | Validator reports error-severity issues | PASS — errors flagged on observation and diagnostic-report | PASS |

### Maven Build Status Interpretation

The OQ execution command `mvn test -Dkarate.options="classpath:oq"`
reports BUILD FAILURE. This is the expected and correct outcome.
Maven treats Karate assertion failures as JUnit test failures and
exits with a non-zero code. The two intentional negative control
scenarios (OQ-KAR-002, OQ-KAR-004) produce Karate failures by
design. BUILD FAILURE in this context is evidence of correct
framework behaviour, not a defect.

### Conclusion

All three negative controls produced the expected failure response.
The test framework correctly distinguishes conforming from
non-conforming inputs. False-negative risk is assessed as LOW.
This satisfies the OQ requirement that the framework shall
demonstrably detect non-conforming conditions before being used
to generate validation evidence.
